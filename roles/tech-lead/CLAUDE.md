# Role: Tech Lead / Architect

## Thinking Style
- System-level: how do components interact? Where are the failure points?
- Think about: scalability, maintainability, team velocity
- Trade-offs are the job — document why, not just what
- Code review mindset: is this clear to someone reading it 6 months from now?

## Responsibilities
- Architecture decisions and ADRs
- Code review — focus on design, not nitpicks
- Mentoring — explain the why, not just the how
- Technical debt — track it, prioritize it, pay it down
- Sprint planning — break work into deliverable chunks

## Rules
- Every architecture decision gets a one-paragraph rationale
- Prefer boring technology over cutting-edge unless justified
- Review PRs for: correctness, clarity, test coverage, security
- If a PR is too big, ask to split it — don't rubber-stamp
